# Show, Don't Tell

## What it is
A rule that every post must reference a real system Riché has built or worked with — not an abstract argument about systems in general. Grandstage. Helm Labs. Suzy Decision Engine. IBM personalization. Context Layer Engine. Claude Code Auto Dream. Letta. The 2003 Boston Beer Microsoft Access platform. The named system grounds the post in something the reader can examine, demo, or read more about. Without it, the post is commentary.

## Why it matters
Show-don't-tell is the difference between a thought leadership post and a manifesto. Manifestos are easy to write and easy to ignore. Showing a real system makes the argument testable — "here is the architecture; here is what changed; here is the metric that moved." Readers can disagree with the conclusion, but they can't dismiss it as theory. This is also what separates Riché from AI commentators who critique without building. Every named system is a quiet credential.

## How to apply

1. **Pick one system per post (sometimes two for deep dives).** The system anchors the post. Other systems can be referenced in passing, but one carries the weight.

2. **Use only canonical systems.** From the inventory: Grandstage, Helm Labs, IBM (personalization platforms), Suzy (Decision Engine, 350+ enterprise customers), Context Layer Engine, Claude Code Auto Dream, Letta (when used as the underlying layer for the Context Layer Engine), and the 2003 Microsoft Access platform for Boston Beer.

3. **Tie the system to the argument.** Don't drop the name and move on. Connect it: "At Grandstage, the v1 system retrieved 50 documents. The v2 system retrieved 4. Same data, different decision about what to keep."

4. **Show specifics about the system.** A named system without specifics is still abstract. Add a number, a decision, or a moment.

5. **Reference, don't relive.** A short, dense connection (15-40 words) is enough. Long retellings of past projects make the post about Riché, not about the argument.

## Examples

**Show:** "At Suzy, the Decision Engine launched in six weeks with three pillars — Intelligence, Insights, Impact — that mapped to the consolidation, prioritization, and storage steps of the pipeline. Within the quarter, 350+ enterprise customers were running on it."

**Tell (failed):** "AI products often benefit from staged context architectures. Many companies have seen success with phased pillar-based approaches."

**Show in a Tuesday Signal:** "Anthropic's longer context windows reward teams who already curate well. At Grandstage, we built around small windows three years ago — the architecture still wins under bigger windows because the bottleneck was never window size."

**Show in a Friday Story:** "The Microsoft Access platform I built for Boston Beer in 2003 was a context system before I had the language for it. It had to decide which production runs to surface to the floor manager out of thousands. I called it a database. Twenty-two years later I'm still solving the same problem with better tools — Letta instead of Access, agents instead of queries."

**Show in a Thursday Framework:** A framework post on the Decision-Readiness Test closes with: "Suzy's pipeline ran the test on every signal. The 4x context-size reduction came from removing decision-irrelevant content, not better embeddings."

## Anti-patterns
- Generic system references. "I've worked on AI products that..." Reads as anonymous. Name the product.
- Inventing a system. If Riché hasn't built or worked on it, it's not on the list. Inventing a system to fit a point breaks the entire credibility model.
- Name without connection. Dropping "Grandstage" in a sentence without explaining what about it grounds the argument is name-dropping, not showing.
